# 0.4.0 — Versioned SR-benchmark & data system (DRAFT for review)

**Status:** DRAFT, design-only, no code. Target: `symbolic-data 0.4.0` (+ a `srbf` consumer bump).
**Owner:** symbolic-data (the data layer). srbf consumes; flash-ansr consumes for training data.
**Supersedes for benchmarks:** the 0.3.0 package-data loaders (which become one layer of this model — see §9).

> Drafty on purpose: the goal is to lock the *shape* and surface risks (see §12) before any implementation.

## 0. Review verdict (advisor + 5-lens workflow `wf_0b462b11`) — READ FIRST

The *shape* (versioned spec → problem-set/grammar → raw; symbolic-data owns, srbf consumes) is sound.
But the review found the draft's "unify generate + set under one spec/sampler/source" premise is **largely
unrealizable against the current code**, and the high-value core is much **narrower** than the full vision.
Five lenses + the advisor converged. Two consequences dominate:

**(I) Carve the program in two — this is the headline.**
- **Benchmark-versioning MVP** (set-side only): a thin spec *header* + a manifest/resolver, then HF-publish
  the **already-shipped 0.3.0** problem-sets as versioned artifacts. Delivers ~all the stated headline value
  (versioned, reproducible, sweep-able canonical benchmarks) and **sidesteps every landmine below**. This is
  the recommended 0.4.0.
- **Generate-unification** (lift `ProblemSource`, swap flash-ansr training, reconcile samplers): low
  *incremental* value (training already works), high risk, and **blocked by a hard prerequisite** — see (II).
  → separate, later, optional track. Not 0.4.0.

**(II) Load-bearing premises in the draft that are FALSE against the working tree (must fix before building the generate side):**
- **flash-ansr does not use symbolic_data — it's a byte-identical FORK.** `flash_ansr/expressions/skeleton_pool.py`
  == `symbolic_data/skeleton_pool.py`; `flash_ansr/src` imports `symbolic_data` nowhere; `requirements.txt`
  pins simplipy, not symbolic-data. So there are **three forks of the data layer** (flash_ansr/expressions,
  symbolic_data, srbf's standalone `FastSRBBenchmark`). *(Nuance: the published flash-ansr 0.7.0 carved to
  symbolic_data via shims; the local `lab` training tree under review is the pre-carve 0.4.x fork — and that
  fork is what training actually runs.)* **Dedup is a prerequisite to any generate-unification.**
- **The set path does NOT share `samples.py`.** `FastSRBSource` does its own support/validation split + noise,
  **does not noise validation**, and uses a different RNG draw (`.normal` vs samples.py `.standard_normal`). So
  §6's "split/noise already shared, only X-sampling differs" is wrong — there are **three** divergences
  (X-sampling, split, noise) plus an **RNG-model** divergence (set threads a `Generator`; generate uses
  **global `np.random`**), which also breaks the §3 `seed:` determinism claim.
- **The unified "per-variable-else-global" sampling rule is unrealizable.** Set and generate use **disjoint
  config vocabularies** (`sample_range`/`sample_type[uni|log|int, sign]` vs `dist/range` + nested-resampled
  range / scale / quantize / mixture) with **no shared keys**, so "override" can't be expressed; and the set
  path has **no global-x fallback** (a missing `vars.vN` is a hard `KeyError`, not inheritance).
- **Option (a) "one shared sampler" is a mirage** — it's a tagged union of disjoint field-sets, not one config.
  Honest end state = the `ProblemSource` interface + the (only-generate-side) shared post-processing + **two**
  x-samplers. Drop (a) as the target.
- **The `ProblemSource` lift isn't a lift.** `SkeletonDatasetSource` hard-imports `flash_ansr.data`
  (FlashANSRDataset/preprocessor/tokenizer/padding) → moving it into symbolic-data is a **circular dep**; it
  yields an eval-baked `EvaluationSample`, and the set path builds `EvaluationSample` **directly from a dict —
  never a `symbolic_data.Sample`** (so "both yield `Sample`" is new work + a faithfulness question: what is a
  benchmark's `skeleton`/`constants`/`n_variables_used`?). Also `EvaluationDataSource` is a `Protocol`, not an
  ABC; `SkeletonDatasetSource` is **sequential-only** (the random/iterate mode lives in the set path);
  `load_benchmark` returns a benchmark object, not a `ProblemSource`.
- **Spec-schema footguns:** one `n_points` can't encode support+validation counts; one `noise` means different
  things per kind; **sweeps-as-list-values collide** with already-list fields (`sample_range`, `sample_type`)
  → sweeps must be a separate, named, scalar-only axis block; `mode` (consumption order) is conflated with
  `method` (random vs **grid/`range`** layout — and grid, the canonical Nguyen choice, has no slot);
  `constants:` is generate-only (silently inert on set specs).
- **Versioning/provenance is overstated.** The existing `simplipy-assets` manifest has **neither** sha256 **nor**
  name@version (both net-new); sr-data **already** ships a *different* HF resolver (`_resolve_spec` via git
  `revision`) — two incompatible mechanisms to reconcile; `raw` is "not loaded" yet "pinned with sha256" =
  provenance theater unless verified; the generate path is **not reproducible** (global RNG + PYTHONHASHSEED-
  ordered pickled skeleton `set`s); inherited resolver bugs (`rmtree` of the cache **root** on partial fetch;
  bare-local-path shadowing); the 0.3.0 problem-set YAMLs are **bare mappings with no header home** (wrapping
  them breaks `SpecBenchmark`). → "the chain reconstructs the data" should be softened to "documents the
  inputs"; pick ONE resolution primitive (git-revision/content-hash) and stamp the resolved hash into the
  *result* at resolve-time.
- **Leak-safety is missing and is first-class** (advisor): the design must state how a `generate` source
  *excludes* the `set` benchmarks (contamination). The shared "what is a benchmark" registry that enables this
  is, ironically, the strongest real reason to ever unify the two sides.

**Revised recommendation:** ship the **benchmark-versioning MVP** as 0.4.0 (header schema + one resolver model
+ publish the existing 3 sets + v23.x val as versioned artifacts; reconsider intra-artifact vs cross-artifact
references — the cross-artifact resolver only earns its weight if one problem-set is reused by many specs in
the near term). **Defer** the generate side behind its dedup + determinism prerequisites. **Drop** the unified
single-sampler / single-rule goal; keep "one `ProblemSource` *interface*, two samplers." **Add** leak-safety as
a first-class requirement. The original §1–§12 below are kept for context but are **superseded by this section
wherever they conflict** (especially §4 unified rule, §5 lift, §6 "already shared", §10 staging order).

## 1. Goal

One uniform, versioned way to specify **and reproduce** any SR dataset — fixed benchmarks (FastSRB,
Feynman, Nguyen, the v23.x val set) *and* on-the-fly training generation (Lample–Charton) — such that:

- the **one canonical** dataset is fully specified and usable as-is ("headline results"), and
- the same source is a **basis for sweeps** (vary n_points / noise / range) for scaling experiments,
- every artifact is **versioned**, and a result's version chain reconstructs exactly how its data was made,
- **symbolic-data owns** the format + sampling classes + resolver; **srbf just consumes** them; flash-ansr
  uses the same machinery for training data.

## 2. Three layers, each independently versioned

```
raw            original upstream file(s), verbatim         provenance only, not loaded
   ▲ (generated from)
problem-set    FastSRB-style YAML: id -> {prepared, vars}  the standard problem content   (set sources)
grammar        operator-space + structure distributions    the generator config           (generate sources)
   ▲ (referenced by)
spec           header: source + global sampling + metadata thin usage config; pins a problem-set@v / grammar@v
   ▲ (pinned by)
manifest       name -> {versions, sha256, source URLs}     the resolver index
```

A `set` spec references a **problem-set@v**; a `generate` spec references a **grammar@v**. Symmetric:
the spec is always a thin header that references its versioned "source content". `raw` is optional
(present for upstream-sourced benchmarks; absent for authored grammars).

The reference chain **is** the provenance: a result stamps `spec@v` → pins `problem-set@v` (or
`grammar@v`) → pins `raw@v`, each with a sha256 in the manifest.

## 3. The uniform spec

```yaml
# spec/feynman-canonical.yaml        (a SET source — benchmark)
metadata: {name: feynman, version: 1, license: "facts; FSReD (Udrescu & Tegmark 2020)"}
source:   {kind: set, mode: iterate}      # mode in {iterate, random}; SET only
problems: feynman@1                        # versioned reference -> problem-set
sampling: {n_points: 100, noise: 0.0}      # GLOBAL scalars; x is per-problem in the referenced set
```

```yaml
# spec/train-v23.yaml                (a GENERATE source — training)
metadata: {name: train-v23, version: 1}
source:   {kind: generate, seed: 0}        # seeded stream; NO consumption mode
grammar:  lc-v23@1                          # versioned reference -> grammar config
sampling: {x: {dist: uniform, range: [-10, 10]}, constants: {dist: ...}, n_points: 256, noise: 0.01}
```

```yaml
# spec/feynman-npoints-sweep.yaml    (sweep = same problem-set, varied global knob)
source:   {kind: set, mode: iterate}
problems: feynman@1
sampling: {n_points: [32, 128, 512, 4096], noise: 0.0}
```

- `source.kind ∈ {set, generate}` decides whether a `problems` or a `grammar` reference is present, and
  whether a consumption `mode` applies (`set` → `{iterate, random}`; `generate` → seeded stream, no mode).
- `sampling:` is the **global** policy/defaults. Sweeps = a base spec + list-valued sampling overrides.

## 4. Sampling resolution (the per-skeleton vs global rule)

Effective sampling for `(skeleton, variable)` = the **per-skeleton/per-variable** spec if present, else the
**global** policy. Concretely:

- **set** problem-sets carry per-problem `vars` (FastSRB-style: `vars.v1 = {sample_range, sample_type}`),
  which are variable-*specific* and override the global. The global `sampling` for a set typically only
  carries scalars (n_points, noise) + optional shared defaults.
- **generate** has no per-skeleton data (skeletons are synthesized), so `sampling` is the whole policy and
  must be variable-*agnostic* ("every variable ~ this distribution") + a `constants` distribution.

So the global `sampling.x` is "the default distribution any unspecified variable inherits"; `vars.vN` is the
per-variable override. Same rule at the variable grain.

## 5. The `ProblemSource` ABC (lift + unify what srbf already has)

srbf **already** has this pattern in `srbf/eval/data_sources.py`:
`EvaluationDataSource` (ABC) ← `SkeletonDatasetSource` (pool/generate; already delegates to
`symbolic_data.sample_from_skeleton`, already supports random/iterate via `_apply_sample_strategy`) and
`FastSRBSource` (spec/set). 0.4.0 = **lift the ABC + both impls down into symbolic-data** as `ProblemSource`
(it produces model-agnostic `symbolic_data.Sample`s), and let srbf keep only its eval-specific
`EvaluationSample` wrapper around them.

```
symbolic_data.ProblemSource (ABC)        yields Sample; .prepare(); .__iter__(); .state_dict()
├── SetProblemSource(spec)               enumerated problem-set; mode = iterate | random
└── GenerateProblemSource(spec)          Lample–Charton grammar; seeded stream
```

- `ProblemSource.from_spec(spec)` builds the right impl from `source.kind`.
- `load_benchmark(name)` → resolve `name` via the manifest → fetch artifact(s) → `SetProblemSource`.
- flash-ansr training swaps its bespoke streaming for a `GenerateProblemSource` over `lc-<grammar>@v`.

## 6. ⚠ The core risk: two (X,y) samplers to reconcile

Today the two paths sample `(X, y)` with **different machinery**:

- **set / FastSRB** (`SpecBenchmark`): `simplipy.parse(prepared)` → `code_to_lambda`, with per-variable
  `_sample_points` (`uni`/`log` × `pos`/`neg`/`pos_neg`, integer) from `sample_range`/`sample_type`.
- **generate / pool** (`SkeletonPool.sample_data` via `sample_from_skeleton`): the support-sampling pipeline
  (`support_sampling.py`) + constant sampling (`prior_factory`/`distributions`) + the support/validation
  split + noise in `samples.py`.

Unifying them under one `sampling` spec is the main implementation question. Options:

- **(a) One shared sampler** both kinds configure — the cleanest end state, but requires expressing the
  FastSRB per-variable `sample_type` (log/sign/int) *and* the pool's support sampler in one config + one code
  path. Risk: behavior drift vs the 0.3.0-verified FastSRB/Feynman/Nguyen numbers.
- **(b) Two samplers behind the `ProblemSource` interface** (status quo, just unified at the interface): set
  uses the SpecBenchmark sampler, generate uses `sample_from_skeleton`. Lowest risk, less "uniform" inside.
- **(c) Converge gradually**: ship (b) under the unified spec/ABC now; migrate to (a) once the sampler
  configs are proven equivalent (oracle-gated, like 0.3.0).

Leaning **(c)**. The split/noise/`Sample` construction in `samples.py` is *already* shared; only the
"skeleton → raw (X,y) points" step differs.

## 7. Versioning, manifest, resolver

- References are human-facing `name@version`; the **manifest** maps each to a concrete HF artifact + sha256.
  A version is immutable once published (forward-only, like the simplipy/symbolic-data/flash-ansr chain).
- Resolver lives in **symbolic-data** (generalizes the `simplipy-assets` manifest pattern simplipy already
  uses); fetch + cache + sha256-verify; offline-friendly (cache hit) and package-data fallback for built-ins.
- srbf calls the resolver; it owns no data.

## 8. Config system

One schema serves training and eval — a spec *is* the config. Training = a `generate` spec consumed as a
stream; benchmark eval = a `set` spec consumed `iterate`/`random`. Sweeps = base spec + override doc. This
replaces the ad-hoc `dataset.yaml` (`{skeleton_pool, padding}`) configs.

## 9. Continuity with 0.3.0 (additive)

- The shipped `data/{fastsrb,feynman,nguyen}.yaml` **are** the problem-set layer (§2). 0.4.0 adds the spec
  headers, the `raw/` provenance copies, the manifest, and HF-versioning around them.
- `SpecBenchmark` becomes the `SetProblemSource` sampler (or feeds option (a)); `load_benchmark` keeps working
  (resolves to a `SetProblemSource`). `FastSRBBenchmark` stays a back-compat alias.
- The v23.x **val** set becomes a `set` problem-set artifact (its skeletons exported to FastSRB-style YAML).
- Defer: soose-nc and other sets; the MIA/contamination audit; the symlog-uniform canonical holdout re-bless
  (separate deferred thread, but its grid is a `set`-source-adjacent artifact).

## 10. Staging (proposed)

1. **Format + resolver** (symbolic-data): the spec schema + manifest/resolver + `name@version` resolution;
   wrap the 0.3.0 problem-sets with spec headers; `load_benchmark` resolves through it. (No sampler change.)
2. **`ProblemSource` ABC** in symbolic-data: lift `EvaluationDataSource`/`SkeletonDatasetSource`/`FastSRBSource`
   down, yielding `Sample`; srbf wraps in `EvaluationSample`. Option (b)/(c) two-sampler interface.
3. **grammar artifact + `GenerateProblemSource`**: promote `expression_space.yaml` to a versioned grammar;
   flash-ansr training consumes it.
4. **v23.x val** exported as a `set` artifact; HF-publish the artifacts + manifest.
5. (later) **shared sampler** (option a), oracle-gated against the 0.3.0 numbers.

## 11. Open questions (for the review + advisor)

- Sampler reconciliation: (a) vs (b) vs (c) in §6 — and can a single config express both FastSRB per-variable
  `sample_type` and the pool support sampler without behavior drift?
- ABC placement: does `ProblemSource` belong in symbolic-data with srbf keeping only `EvaluationSample`, or is
  the eval/data boundary drawn elsewhere?
- Reference granularity: are problem-sets / grammars their own HF artifacts referenced cross-artifact, or
  bundled per dataset? (Leaning independent, per the "every level versioned" decision.)
- Version semantics: integer `@v` + sha256 pin — enough? Do we need ranges / "latest"? (Lean: pinned-only.)
- Is this over-built for the near-term need (Feynman/Nguyen/FastSRB + v23.x val + LC training)? What is the
  minimal subset that delivers the headline value?
- Generation determinism across machines (the `feedback_seeds_purely_random` constraint) under a versioned
  `generate` spec.
